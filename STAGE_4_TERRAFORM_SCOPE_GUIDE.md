# Stage 4 - Terraform Scope Definition (Phase 5, Step 1)

This step defines what Terraform owns before writing any `.tf` resources.

## Goal of this phase

Use Terraform as the infrastructure-definition layer for reproducible platform prerequisites, not as a duplicate deployment system.

## Ownership boundary

Terraform owns:

1. Namespace foundation
   - `music-platform` namespace lifecycle.
   - Baseline namespace labels/annotations required by platform behavior (for example `istio-injection=enabled`).
2. Platform-level prerequisites
   - Cluster resources that must exist before app deployment starts.
   - Shared environment-level settings consumed by deployment automation.
3. Deployment dependency resources
   - Infrastructure pieces that GitHub Actions and Helm rely on (for example namespace and foundational policies).

Terraform does not own:

1. Application release objects generated by Helm
   - API Deployment/Service and DB StatefulSet/Service from `helm/music-platform`.
2. Day-to-day app version rollouts
   - Image tag updates and release upgrades remain in CI/CD (`helm upgrade --install`).
3. Runtime traffic/security policy tuning done as application delivery
   - Existing Istio manifests remain in `k8s/istio` unless explicitly moved later.

## Why this boundary is safe

- Avoids Terraform/Helm ownership conflicts on the same Kubernetes objects.
- Keeps app deployment fast in CI/CD while Terraform manages slower-changing infrastructure baseline.
- Makes responsibility explicit:
  - Terraform = prerequisites and platform baseline.
  - Helm = application release packaging.
  - GitHub Actions = automated build/deploy execution.

## Initial target set for Terraform (Step 1 decision)

The first Terraform implementation should target only:

1. `kubernetes_namespace` for `music-platform`.
2. Namespace labels/annotations required for Istio injection behavior.

Future expansion can include additional platform prerequisites after this baseline is stable.
